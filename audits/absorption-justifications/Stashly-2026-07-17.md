## Why ARCHIVE_ONLY instead of `crates/stashly`

The task asked to absorb Stashly into `pheno/crates/stashly` as a "stash/secret storage" crate. Preflight audit surfaced a hard mismatch on the description, plus two domain conflicts that would have created real damage in the workspace:

1. **Domain mismatch.** `Stashly/Cargo.toml:1` declares the crate as a **caching** framework. `Stashly/src/lib.rs:1` opens with "Stashly — Universal caching framework with multi-tier and singleflight support." It has already absorbed `thegent-cache` upstream. Calling it secret-storage is wrong.

2. **Capability already covered in pheno.** `pheno` already ships two caching crates:
   - `crates/agileplus-cache` — caching domain
   - `crates/phenotype-cache-adapter` — caching adapter domain
   Adding `crates/stashly` would create a 3rd cache crate with overlapping scope.

3. **Duplicate `phenotype-cache-adapter` package name.** `Stashly/src/phenotype-cache-adapter/Cargo.toml` declares `[package] name = "phenotype-cache-adapter"`. Cargo forbids two workspace members from declaring the same package name. Even a partial lift would have required either a rename or surgery on the inner crate.

4. **Empty upstream `tests/`.** `Stashly/tests/` contains only a README; there were no integration tests to copy in.

5. **Registry id was wrong in the task.** Task said `id=repo-Stashly`; the actual `disposition-index.json` row id is `repo-Stashly` but the `catalog/registry.yaml` row id is `stashly`. The literal task would have updated zero rows.
