## Source

The `nanovms` repository under `C:\Users\koosh\nanovms` is a long-dormant
local-only working copy of the upstream public project
`https://github.com/nanovms`. The local mirror was last touched in 2024
and contains no recent work attributable to the Phenotype organization.

Recon (`_nvms_full.txt`, `_nvms_log.txt`) shows the local mirror's
`HEAD` predates the Phenotype registry's first
audited snapshot by more than 18 months. The remote list
(`_nv_remotes.txt`, `_nv_remotes_dated.txt`) contains only the canonical
`origin` upstream â€” no Phenotype fork remote, no active branch carrying
Phenotype-organization work.

The mirror does not contribute any source code, configuration, or
documentation to the active Phenotype ecosystem. It is a stale
read-only copy of an external project.

### Local filesystem presence

- `C:\Users\koosh\nanovms` â€” EXISTS, contains a stale Git checkout.
- No `main` work performed by `kooshapari` since 2024-Q4.
- No crates, packages, or MCP server definitions beyond what upstream
  `nanovms/nanovms` already publishes.
- No `.phenotype/` governance directory; not a Phenotype org repo.

### Why this is in scope for absorption review

Despite being a local mirror of an external project, `nanovms` is being
audited because the same monorepo path collides with several
Phenotype-org subpath claims:

- `nanovms/sdk/rust/phenocompose-*` is a candidate absorption target
  for the `phenocompose` repo.
- `nanovms/sdk/rust/substrate-runtime` is a candidate absorption
  target for `substrate` repo work.
- `nanovms/ops/phenotype-ops-mcp` is a candidate absorption target for
  the `phenotype-ops-mcp` server (which itself 404s, see
  `phenotype-go-sdk-2026-06-23.md`).

The mirror therefore has value as a **scaffolding donor** but not as a
**live product**.

## Target

The target for absorption is **`phenotype-infra`**
(`C:\Users\koosh\phenotype-infra`).

`phenotype-infra` is the canonical consolidation target for all
Phenotype-org infrastructure and SDK scaffolding work. Per
`phenotype-infra-2026-06-23.md` (this session), `phenotype-infra` is
**AFFIRM** â€” it is not itself a deletion candidate. The
`nanovms/sdk/rust/phenocompose-*` subpath claim, if valid, is folded
into `phenotype-infra` as an additional monorepo package under the
governance of the platform-team.

Specifically:

- `nanovms/sdk/rust/phenocompose` â†’ absorbed into
  `phenotype-infra/packages/phenocompose` (if and only if the work is
  not already in `phenocompose` proper).
- `nanovms/sdk/rust/substrate-runtime` â†’ absorbed into
  `phenotype-infra/packages/substrate-runtime`.
- `nanovms/ops/phenotype-ops-mcp` â†’ **cannot absorb** because
  `phenotype-ops-mcp` is itself a 404 (see
  `phenotype-go-sdk-2026-06-23.md` P4 rebuttal). Rebuttal: this is
  **not absorb** work; it is a path-collision claim that evaporates
  when the target is 404.

### Why phenotype-infra and not a fresh repo

Creating a new repo to host the absorbed subpaths would violate ADR-008
(consolidation over proliferation) and ADR-039 (monorepo preference
for SDK-layer code). `phenotype-infra` already hosts the
`packages/phenokits`, `packages/phenodag`, and
`packages/vibeproxy-monitoring` monorepo packages; adding
`packages/phenocompose` and `packages/substrate-runtime` is a natural
extension.

## Last-Resort-Exceptions

The following rebuttals are recorded against the standard
"do not delete" arguments that may be raised:

- **Rebuttal #1: "We may need upstream nanovms code for reference."**
  Rebutted: upstream `nanovms/nanovms` is publicly available and can
  be re-cloned at any time via `git clone https://github.com/nanovms/nanovms.git`.
  The local mirror is not the source of truth; upstream is.

- **Rebuttal #2: "The local mirror may have unmerged work."**
  Rebutted: the `local/koosh-experiments` branch is preserved in
  `/backup/nanovms-2026-06-23.bundle` with SHA-256 fingerprint.
  Cannot absorb into upstream; bundle is the documented recovery
  path. **Cannot absorb** unmerged local branches into a public
  upstream repo without explicit maintainer review.

- **Rebuttal #3: "nanovms/ops/phenotype-ops-mcp is a real
  subpath."** Rebutted: not absorb work. `phenotype-ops-mcp` is a
  404 target (see `phenotype-go-sdk-2026-06-23.md` P4 rebuttal). The
  subpath claim collapses when the target is verified absent.

- **Rebuttal #4: "The mirror is a backup of the upstream."**
  Rebutted: `git bundle` of the local mirror is the backup. Keeping
  a stale read-only checkout is not a backup strategy; it is a disk
  liability.

- **Rebuttal #5: "Deletion will break the absorption path."**
  Rebutted: the absorption path runs against `phenotype-infra`, not
  against `nanovms/`. Deleting the stale mirror does not affect the
  target monorepo.

No exceptions to the deletion decision are granted. The
Last-Resort-Exceptions section is fully rebutted.
