## Executive Summary

The original absorption plan for `KooshaPari/DataKit` (batch4, row
`repo-DataKit-batch4`) was constructed from **stale telemetry** that
mis-classified DataKit as a "Rust BLOCK A app". Actual source audit
contradicts the row on **four independent axes**:

1. **Language** — source is **Python** (`pyproject.toml`,
   `[tool.hatch.build.targets.wheel] packages = ["src/datakit"]`),
   not Rust. No `Cargo.toml`, no `package.json` exists at repo root.
2. **Status** — repo is **DEPRECATED — ARCHIVE RECOMMENDED** per
   its own `README.md` (line 1–4) and `TOMBSTONE.md` (status line 3).
3. **Prior absorption** — DataKit was already absorbed into
   `phenotype-python-sdk/packages/data-kit` per commit `d08cf10`
   ("DataKit fully absorbed into phenotype-python-sdk (#53)"),
   referenced in `TOMBSTONE.md` line 8.
4. **Canonical placement** — `RATIONALIZATION_EXECUTION.md` §2
   (line 76, 80) designates `phenotype-python-sdk` as the
   target; `phenotype-python-sdk/packages/data-kit/` is the
   active, populated landing pad (Python `db_kit/`, Rust
   `Cargo.toml` + `phenotype-event-bus/` + `phenotype-in-memory-store/`,
   Go `go.work`).

Therefore the registry row `repo-DataKit-batch4` is corrected in
place: `disposition` flips from `ABSORB` to `ARCHIVE_ONLY`, the
target is corrected to the canonical `phenotype-python-sdk/packages/data-kit`,
and the `phenotype-apps/apps/datakit/` target is **voided**.

The `pheno-archive` boundary is the correct placement: the source
GitHub repo is archived (read-only) so history is preserved, no live
imports exist (verified by fleet-wide grep), and the canonical
implementation continues to evolve in
`phenotype-python-sdk/packages/data-kit/`.

   Rebutted: the canonical implementation **already exists** in

   merge. Archiving is the correct terminal state.
