## Why ARCHIVE_ONLY instead of `pheno/crates/configra/`

The task spec called for absorption into `pheno/crates/configra` if the source is library-shaped. Configra is lib-shaped (5 libs, 1 bin) and the source is healthy. **However, the literal copy would create three concrete conflicts** that match the Stashly-2026-07-17 failsafe conditions. The disposition is therefore **ARCHIVE_ONLY** per the task's own fallback clause: "Failsafe: ARCHIVE_ONLY if conflicts."

### Conflict #1 — Duplicate `phenotype-config-loader` package name

- `Configra/crates/phenotype-config-loader/Cargo.toml:2` declares `[package] name = "phenotype-config-loader"`.
- `pheno/crates/phenotype-config-loader/` already exists with the **same** package name (verified 2026-07-17, `pheno/crates/phenotype-config-loader/CANONICAL.md:1-23` is present).
- Cargo forbids two workspace members from declaring the same package name. Even a partial lift would require either a rename or surgery on the inner crate.
- This is the **exact** Stashly-2026-07-17 failsafe pattern (`Stashly/src/phenotype-config-adapter/Cargo.toml` had the same problem).

### Conflict #2 — Domain already covered in `pheno`

`pheno` already ships config-domain crates that overlap with each of the four Configra sub-crates:

| Configra sub-crate | Existing pheno equivalent |
| --- | --- |
| `pheno-config` (typed `Config` + `ConfigBuilder` + env-var cascade) | `pheno/crates/phenotype-shared-config` (config types, `Cargo.toml:1-25`); `pheno/crates/phenotype-flags` (flag management) |
| `phenotype-config-loader` (generic JSON/TOML loaders) | `pheno/crates/phenotype-config-loader` (legacy, `CANONICAL.md` already declares it superseded by Configra upstream) |
| `config-schema` (field-shape validator) | `pheno/crates/phenotype-validation` (validation primitives) |
| `settly` (settings lifecycle + encryption + hot-reload) | `pheno/crates/phenotype-state-machine` + `pheno/crates/phenotype-casbin-wrapper` (settings/policy) |
| `configra-ops` (observability + health) | `pheno/crates/phenotype-telemetry` + `pheno/crates/phenotype-observability` |

Adding 5 more Configra sub-crates would create a 6×+ config domain in the workspace. The existing `pheno/crates/phenotype-config-loader/CANONICAL.md` already documents that `KooshaPari/Configra` is the **canonical home** (ADR-031) — the absorption would reverse-engineer an already-canonicalized repo back into a duplicate.

### Conflict #3 — Workspace dependency bloat

`Configra/crates/settly/Cargo.toml:34-49` pulls in heavy 3rd-party deps that pheno's workspace doesn't currently carry:

- `aes-gcm = "0.10"` (encryption-at-rest)
- `argon2 = "0.5"` (KDF)
- `sqlx = { version = "0.8", features = ["runtime-tokio", "postgres", "chrono", "uuid"] }` (Postgres)
- `tokio-postgres = "0.7"` (Postgres client)
- `redis = { version = "1.2", features = ["tokio-comp", "connection-manager"] }` (Redis)
- `validator = "0.18"` (derive validation)
- `notify = "6.1"` (file-watch)

These would add **30-90s** to the pheno workspace cold-compile and create a new 3rd-party surface (postgres/redis/sqlx) that the rest of pheno doesn't depend on. The `pheno-config` sub-crate alone transitively pulls these in via `path = "../settly"` if any consumer of the Configra copy also uses `settly`.

### Conflict #4 — `configra-ops` is half CLI, half lib

`Configra/crates/configra-ops/Cargo.toml:17-19` declares both a lib (`configra_ops`) and a `[[bin]] name = "configra-ops"`. The task spec puts CLIs into `phenotype-tooling`, not `pheno/crates/`. A partial split (lib → pheno, bin → phenotype-tooling) would require renaming or duplicating the package — same anti-pattern as the Stashly failsafe.

## Reactivation path (for future agents)

If a future agent wishes to re-attempt absorption (e.g. to a NEW target that doesn't collide with pheno):

1. Unarchive on GitHub: Settings → Danger Zone → Unarchive (or `gh repo unarchive KooshaPari/Configra` if the account still has the right).
2. Pick a target without the conflicts. Candidate: a fresh standalone `phenotype-configra` repo, or the existing `phenotype-tooling` workspace (which already has `phenotype-tooling/crates/phenotype-config/` for the same role and would also need a re-collision check).
3. Resolve package-name collisions first. If the target is `pheno`, rename `phenotype-config-loader` → `phenotype-config-loader-v2` (or deprecate the legacy one in `pheno/crates/phenotype-config-loader/` per its own `CANONICAL.md`).
4. Drop the heavy 3rd-party deps that are out-of-scope for a config substrate (sqlx/redis/tokio-postgres → extract to a separate `phenotype-encryption` crate if needed).
5. `cargo check --workspace` in the target repo; resolve any other collisions.
6. Update the registry row back to `disposition: ABSORB` with the new target + relocation evidence.
