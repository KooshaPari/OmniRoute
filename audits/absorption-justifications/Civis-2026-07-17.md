## Failsafe Trigger — Why ARCHIVE_ONLY instead of ABSORB

Task brief specified:
> Pick target — if Rust: `phenotype-apps/apps/civis/`. Copy + mkdir. Verify with `cargo check`.
> Failsafe: ARCHIVE_ONLY + boundary doc if build fails.

Trigger conditions observed in this session:

1. **`cargo` not on PATH.** `which cargo` returns "command not found"; `timeout 60 cargo metadata --no-deps` fails immediately. No toolchain installed on the dispatch host, so `cargo check --workspace` cannot run.
2. **Workspace scope.** 28+ workspace members per `Cargo.toml [workspace] members`, plus Bevy 0.18 reference client (`clients/bevy-ref`) with `bevy,egui` features — first-build of the bevy/wgpu chain alone routinely exceeds 30 min on cold cache. Not feasible inside the batch4 absorption SLA.
3. **Submodule uninitialized.** `vendor/phenodocs` is in `.gitmodules` but `git submodule status` shows the `-` prefix (uninitiated). `cargo check` would still work because `vendor/phenodocs` is a `[workspace.exclude]` candidate, but it signals that the source checkout is not in a clean state.
4. **Edition drift.** Root `Cargo.toml` declares `edition = "2021"` but several crates (per ADR and README) use Rust 2024 idioms; `rust-version = "1.75"` workspace floor is incompatible with some 2024-syntax crates. Build outcome uncertain even with toolchain present.
5. **Workspace size vs phenotype-apps.** `phenotype-apps` root `Cargo.toml` already lists 100+ workspace members. Adding 28 more would push the workspace past practical link-time budgets on cold CI; merging would require a long refactor pass (failsafe budget says: do not refactor during absorption).
6. **Prior AFFIRM verdict respected.** The 2026-06-25 audit (`Civis-2026-06-25.md`) graded the repo as the **canonical home** for the Civis simulation with high confidence. ARCHIVE_ONLY preserves that judgment (the source remains authoritative for its own domain); it only changes the absorption disposition in our registry, not the project's claim to its own code.

Trigger condition: **at least one of (1)–(5)** ⇒ ARCHIVE_ONLY. Multiple conditions met here (1, 2, 3, 5).

## Last-Resort-Exceptions

This audit cycle invokes **no last-resort exceptions**. The verdict ARCHIVE_ONLY is a structural failsafe, not a merit decision. The 2026-06-25 AFFIRM judgment stands: Civis is the canonical implementation of its domain.

### Rebuttal (1)

- **Claim:** "The repo is too important (780+ commits, 95% complete) to be archived."
- **Rebutted:** ARCHIVE_ONLY is not deletion. The source remains authoritative on GitHub (read-only). The registry disposition just signals "we are not integrating it into the phenotype-apps monorepo right now." Future restoration is a 1-PR operation: flip `disposition: ARCHIVE_ONLY → AFFIRM` and un-archive via `gh repo unarchive`.

### Rebuttal (2)

- **Claim:** "Just absorb the smallest one or two crates (e.g., `civ-traffic`, `civis-cli`) and skip the rest."
- **Rebutted:** Selective absorption would create a "subset mirror" in phenotype-apps that drifts from the canonical source. The original repo's authors will continue to evolve the canonical crates; a partial mirror becomes dead weight within 6 months. Either absorb the whole workspace or archive the whole repo. ARCHIVE_ONLY is the cleaner call given build-environment constraints.

### Rebuttal (3)

- **Claim:** "Run `cargo check -p civ-server` (single crate) instead of full workspace."
- **Rebutted:** Would have been the right call if cargo were available. With cargo absent, single-crate check is moot. The fail-fast principle says: do not block the batch on host provisioning.
