# Pine → substrate (ARCHIVE_ONLY fallback) — 2026-07-17

**Source repo:** KooshaPari/Pine
**Planned target:** substrate (crates/pine)
**Final disposition:** **ARCHIVE_ONLY** (boundary-shape mismatch; not absorbed into substrate)

## Audit summary

Pine is described in its README as a "Wine-equivalent compatibility layer for Phenotype — Windows and cross-platform application translation." Per the registry intake (`phenotype-registry/projects/Pine.json`, `phenotype-registry/registry/disposition-index.json` row `repo-Pine`), it was planned for absorption into the `substrate` monorepo as `crates/pine`. Source size: ~1.5 MB, single remote branch.

## Structural findings (truthful audit)

The original absorption plan ("small Rust crate, copy `Pine/src` + `Pine/tests` + `Pine/Cargo.toml` + `Pine/README.md` into `substrate/crates/pine/`") **does not match Pine's actual layout**.

**Top of repo (`/Users/kooshapari/CodeProjects/Phenotype/repos/Pine/`):**

- `Cargo.toml` — a 3-line workspace manifest (`[workspace] / members = ["crates/*"] / resolver = "2"`), not a single-crate manifest.
- No top-level `src/` directory.
- No top-level `tests/` directory.
- `Cargo.lock`, `Justfile`, `Taskfile.yml`, `deny.toml`, etc. — monorepo scaffolding, not single-crate.

**Actual layout:** `crates/` containing **five** sibling crates:

| Crate         | Purpose (real code / stub)                                                            | Dependencies              |
| ------------- | ------------------------------------------------------------------------------------- | ------------------------- |
| `pine-core`   | Foundation types + traits + port directory; mostly stub, real types are `ProcessId`, `SyscallNumber`; `Loader`, `SyscallHandler` traits; `ports/{assets,resolve,section,serialization}.rs` directories | `thiserror`, `serde`, `serde_json` |
| `pine-compat` | Empty shim (`CompatibilityLayer::new()` only)                                         | `pine-core`               |
| `pine-loader` | Real ELF + PE parser using `goblin` (500-line `lib.rs`, integration tests round-trip a tmp ELF) | `pine-core`, `goblin = "0.10.7"` |
| `pine-nvms`   | Empty shim (`NvmsRuntime::new()` only)                                                | `pine-core`               |
| `pine-syscall`| Real x86_64 Linux syscall number→name table (300+ entries), translator trait, error/result types; extensive unit tests | `pine-core`, `libloading` |

The planned `substrate (crates/pine)` (singular) target therefore cannot be satisfied without flattening the workspace, renaming crates, and resolving intra-Pine `path = "../pine-core"` dependencies. None of that matches the original task instructions.

## Build-conflict assessment

substrate (2026-07-17 HEAD `1c64ef2`) is a Cargo workspace with ~50+ existing crates and `thiserror = "2.0"` pinned in `[workspace.dependencies]`. Pine pins `thiserror = "1"` in `pine-core`. The major-version conflict is technically resolvable (per-crate override) but compounds the flattening overhead and is itself a signal that Pine was never designed for substrate integration.

## Disposition decision

Per the absorption task's own fallback clause ("If Pine has build conflicts or substrate isn't a workspace, ARCHIVE_ONLY + boundary doc + registry update. Be truthful about failures."):

- **GitHub:** `gh repo archive KooshaPari/Pine --yes` executed 2026-07-17 (`isArchived: true` verified).
- **substrate:** No crate added, no commit made. substrate working tree was intentionally left untouched.
- **Registry:** `KooshaPari/Pine` row updated from `ABSORB / fsm=active` → `ARCHIVE_ONLY / fsm=absorbed`, with the rationale appended to the `note` and `archived_date` filled.

This preserves the registry invariant that disposition reflects reality, and prevents a fake "absorbed into substrate (crates/pine)" claim when no such crate exists.

## Reactivation path (for future agents)

If a future agent wishes to re-attempt absorption:

1. Delete `isArchived` flag on GitHub (Settings → Danger Zone → Unarchive).
2. Create branch `pine-reabsorb/wip-2026-XX-YY`.
3. Copy each of `crates/pine-{core,compat,loader,nvms,syscall}/` into `substrate/crates/` keeping names.
4. Override `thiserror = "1"` in each Pine crate's `Cargo.toml` (per-crate, since substrate pins 2.0).
5. Add all 5 to `[workspace.members]` of `substrate/Cargo.toml`.
6. Update `substrate/[workspace.dependencies]` only if/when Pine crates need a centralized version pin.
7. Run `cargo check -p pine-core -p pine-loader -p pine-syscall` (the 3 crates with real code; the 2 empty ones are trivial).
8. Update registry row back to `disposition: ABSORB` with relocation evidence.
