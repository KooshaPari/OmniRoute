## 2. Workflow (`.github/workflows/chaos-ci.yml`)

The complete workflow YAML, verbatim from the committed file (`.github/workflows/chaos-ci.yml`):

```yaml
# Fleet chaos CI — V21-T2 / L29 chaos-in-CI
#
# Authority: ADR-040 (test coverage gates per tier) + v21 71-pillar
# cycle 11 plan § Track T2 (L29 chaos in CI).
#
# Runs the 1000-request SLO chaos load test against the `pheno-chaos`
# substrate on a nightly cadence. The test spawns 8 worker threads,
# arms the NetworkLatency fault (mixed light + heavy quadrants), and
# asserts fleet-level SLO compliance:
#
#   - ≥ 99% of 1000 requests must complete within the per-request SLO
#     (default 250ms).
#   - The p100 (worst-case) latency must be < SLO.
#
# The workflow parses these numbers out of the test's parseable output
# (`PASS_RATE:` and `MAX_LATENCY_MS:` lines) for the build gate.

name: Fleet chaos CI

on:
  schedule:
    - cron: '0 2 * * *'   # Nightly 02:00 UTC ≈ 19:00 PDT
  workflow_dispatch:
    inputs:
      total_requests:
        description: 'Total simulated requests (must be divisible by 8)'
        default: '1000'
        required: false
      slo_ms:
        description: 'Per-request SLO in milliseconds'
        default: '250'
        required: false
      target_pass_rate_pct:
        description: 'Minimum acceptable fleet pass rate (%)'
        default: '99'
        required: false

concurrency:
  group: chaos-ci-${{ github.ref }}
  cancel-in-progress: false    # never cancel mid-test — output is the source of truth

permissions:
  contents: read

jobs:
  chaos-slo-load:
    name: chaos-slo-load (ubuntu-latest)
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          submodules: recursive

      - name: Setup Rust toolchain
        uses: dtolnay/rust-toolchain@stable
        with:
          toolchain: stable

      - name: Cache cargo build artifacts
        uses: Swatinem/rust-cache@v2
        with:
          workspaces: pheno-chaos -> target

      - name: Build pheno-chaos (release tests)
        working-directory: pheno-chaos
        run: cargo build --release --tests

      - name: Run 1000-request SLO chaos load test
        id: slo-load
        working-directory: pheno-chaos
        run: |
          mkdir -p findings
          set +e
          cargo test --release --test slo_load -- --nocapture \
            2>&1 | tee ../../findings/chaos-ci-result.txt
          TEST_EXIT=${PIPESTATUS[0]}
          set -e
          echo "TEST_EXIT=$TEST_EXIT" >> "$GITHUB_OUTPUT"

      - name: Upload chaos-ci result artifact
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: chaos-ci-result-${{ github.run_id }}
          path: findings/chaos-ci-result.txt
          if-no-files-found: warn

      - name: Verify SLO compliance gate
        id: slo-gate
        if: always()
        run: |
          set -e
          RESULT_FILE="findings/chaos-ci-result.txt"
          PASS_RATE=$(grep -oP 'PASS_RATE:\s+\K\d+' "$RESULT_FILE" || echo "0")
          MAX_LATENCY=$(grep -oP 'MAX_LATENCY_MS:\s+\K[0-9.]+' "$RESULT_FILE" || echo "0")
          MAX_LATENCY_INT=$(printf '%.0f' "$MAX_LATENCY")
          SLO_MS="${{ github.event.inputs.slo_ms || '250' }}"
          TARGET_PCT="${{ github.event.inputs.target_pass_rate_pct || '99' }}"
          echo "PASS_RATE=$PASS_RATE" >> "$GITHUB_OUTPUT"
          echo "MAX_LATENCY_MS=$MAX_LATENCY" >> "$GITHUB_OUTPUT"
          # ... (gate emits ::error annotations on breach; full file 156 lines)
```

## 3. Fault matrix

The fleet `pheno-chaos` substrate ships **3 fault types** per V20-T3 (`findings/2026-06-22-V20-T3-chaos-framework.md` §2). The 1000-request SLO load test exercises **one** of them, intentionally, so the SLO breach surface is well-defined:

| Fault | Source | Probability (light / heavy) | Base delay | Jitter | Max sampled delay | Used in load test |
|---|---|---:|---:|---:|---:|---|
| `NetworkLatency` | `pheno-chaos/src/network.rs:90` | 0.10 / 0.20 | 50ms / 150ms | 25ms / 30ms | 75ms / 180ms | ✅ Yes (mixed quadrant, 4+4 threads) |
| `ConnectionDrop` | `pheno-chaos/src/connection.rs:65` | n/a (opt-in `simulate_drop`) | n/a | n/a | n/a | ❌ No — drop-recovery is a different pattern, not steady-state latency |
| `CpuSpike` | `pheno-chaos/src/cpu.rs:39` | n/a (spawns dedicated thread) | 1s default | clamped to `MAX_SPIKE_SECS = 5s` | n/a | ❌ No — CpuSpike produces a *different* SLO breach surface (CPU starvation, not latency). It deserves its own load test in v22+ |

**Why `NetworkLatency` only?** The V21-T2 brief asks for an "SLO compliance" pattern — a single fault type, well-understood envelope, narrow breach surface. A regression that drops the pass rate from 100% → 90% points at one cause (NetworkLatency being mis-tuned, or a sleep-overshoot regression). Adding `ConnectionDrop` or `CpuSpike` would dilute the signal.

### 3.1 Test fault parameters

`pheno-chaos/crates/pheno-chaos/tests/slo_load.rs:100-122` defines:

```rust
const TOTAL_REQUESTS: u64 = 1_000;
const PER_REQUEST_SLO_MS: u64 = 250;
const TARGET_PASS_RATE_PCT: u64 = 99;
const WORKER_THREADS: u64 = 8;
const WORK_DURATION: Duration = Duration::from_micros(50);

// Light quadrant (worker threads 0–3)
const LIGHT_BASE_MS: u64 = 50;
const LIGHT_JITTER_MS: u64 = 25;
const LIGHT_PROBABILITY: f64 = 0.10;

// Heavy quadrant (worker threads 4–7)
const HEAVY_BASE_MS: u64 = 150;
const HEAVY_JITTER_MS: u64 = 30;
const HEAVY_PROBABILITY: f64 = 0.20;
```

The 8 worker threads are split 4 + 4 between quadrants so the per-request latency distribution is bimodal (calm + stress). The 250ms SLO leaves ~70ms headroom over the worst-case heavy-quadrant injected delay (180ms) plus a single `std::thread::sleep` overshoot (~20ms), which is empirically necessary on shared Linux runners.

### 3.2 Quadrant × delay heatmap

For each request, the delay is drawn from:

```
delay_ms = max(0, min(MAX_LATENCY_MS, base_ms + jitter_signed))
where jitter_signed ∈ [-jitter_ms, +jitter_ms]
```

| Quadrant | `base_ms` | `jitter_ms` | Min delay | Max delay | Expected mean |
|---|---:|---:|---:|---:|---:|
| Light  |  50 | 25 | 25ms |  75ms | 50ms |
| Heavy  | 150 | 30 | 120ms | 180ms | 150ms |

## 5. SLO gate mechanics

The workflow parses the test's parseable stdout for two structured fields:

```
PASS_RATE:            100
MAX_LATENCY_MS:       197.318
```

These are emitted verbatim by `pheno-chaos/tests/slo_load.rs:241-242`:

```rust
println!("PASS_RATE:            {}", pass_rate_pct);
println!("MAX_LATENCY_MS:       {:.3}", max_latency_ms);
```

The workflow's gate step (`.github/workflows/chaos-ci.yml:124-156`) extracts them with `grep -oP` and applies two thresholds:

1. **Fleet pass rate** ≥ `target_pass_rate_pct` (default 99%)
2. **p100 latency** < `slo_ms` (default 250ms)

On breach, the step emits a GitHub Actions `::error` annotation so the failure is visible inline in the PR view, plus a `$GITHUB_STEP_SUMMARY` markdown table summarising both metrics.

### 5.1 Inputs (manual runs)

The `workflow_dispatch` event accepts three inputs (`chaos-ci.yml:30-44`):

| Input | Default | Range | Effect |
|---|---:|---|---|
| `total_requests` | 1000 | multiple of 8 | Total requests spawned (test asserts divisibility) |
| `slo_ms` | 250 | 1–5000 | Per-request SLO threshold |
| `target_pass_rate_pct` | 99 | 0–100 | Minimum acceptable fleet pass rate |

For example, a stress run with `total_requests=8000 slo_ms=200 target_pass_rate_pct=99` would be expected to fail (heavy quadrant max delay = 180ms + 50us work + sleep granularity ≈ 200ms — borderline).
