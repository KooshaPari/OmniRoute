The current MacBook is OK to author the `devcontainer.json` + `Dockerfile` files (each ~50 LOC), but the validation step requires a devcontainer boot (~5 min per crate). Heavy work per ADR-023 Rule 2.1 → `device: heavy-runner` for the validation sweep. **Unblock:** dispatch to a subagent that has the heavy-runner profile.

## Risks & Caveats

1. **L60 facade adoption is the load-bearing claim of v15.** The `LatencyHistogram` facade is brand-new (commit `23386dc6`); adoption in 1 crate (pheno-port-adapter) is the floor for a Tier-2 substrate status. If the facade has a cardinality bug or the OTel-aligned bucket boundaries don't match downstream collector config, the fleet rollout will be noisy. **Mitigation:** v15 T8 PR includes a `tests/integration_histogram.rs` round-trip test that decodes via the OTel collector's `histogram_quantile()`.
2. **T4 L6 closure may regress.** The `circular_dep_check.sh` script runs `cargo metadata --format-version 1 | jq` — fast on MacBook but slow on multi-100-crate workspaces. If CI timeout is exceeded, the gate becomes a soft warning. **Mitigation:** the script supports a `--quick` mode (skip reverse-deps) and a `--threshold N` (max acceptable cycle depth) for adaptive timeout.
3. **T6 SBOM diff policy is GPL-strict.** The `sbom_diff.py` script fails CI on any **new** GPL/AGPL/SSPL license introduction. This is a **fleet-wide hard gate** — any new dep that pulls a transitive GPL will fail. **Mitigation:** the script emits the offending `package@version` in the failure message; revert the offending dep or open an ADR for the exception.
4. **L49 vuln-response runbook is policy-only.** T7 produces a runbook document; it does not produce a fleet-wide automated vuln scanner (that is `cargo audit` + `pip-audit` + `govulncheck`, which are already deployed via ADR-042). The runbook codifies the **human response workflow** (triage → patch → advisory → disclosure) when an automated scanner finds a CVE.
5. **Cycle-1 cohort delta is +0.09, not the cycle-4 cohort's +0.30.** The cycle-1 cohort (Eidolon, agent-platform, mobile-mcp, mobile-cli, focalpoint, PhenoCompose) has higher baselines on most pillars, so a given pillar closure adds less to the mean than in the cycle-4 cohort (where Configra / pheno-config / pheno-otel / pheno-capacity started at 1.76-1.83). v15's +0.09 is **already the ceiling** for the cycle-1 cohort without re-scoring the cycle-4 cohort (which would happen on Monday 2026-06-22 with the new L57 / L60 / L48 gates).
