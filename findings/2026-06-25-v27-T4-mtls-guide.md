# v27-T4: L52 mTLS Fleet Adoption Guide

**Pillar:** L52 — mTLS between pheno-* substrates (2.0 → 2.5, cycle-17 P0)
**Owner:** orch-v27
**Effort:** 2h (30 min spec + 1.5 h cert rotation impl)

## Why

Without mTLS, any process on the same host (or in the same k8s namespace) can
connect to any pheno-* gRPC/HTTP2 port without authentication. This is
acceptable for dev but fails SOC2 CC6.6 (logical access) and SLSA L3 (provenance).

## What to build

1. **Internal CA key/cert provisioning** — `tools/mtls-ca/provision.sh`:
   - Generate a root CA (10-year validity, stored in Vault at
     `secret/phenotype/pki/ca`)
   - Issue per-service leaf certs (1-year validity) signed by the root CA
   - Leaf certs contain SPIFFE-like SAN: `spiffe://phenotype/<org>/<service>`

2. **gRPC interceptors** — for each service (pheno-flags, pheno-port-adapter,
   pheno-config, pheno-context):
   - Server: verify client cert against CA bundle, extract SAN from cert
   - Client: present leaf cert + private key, verify server cert
   - On failure: close connection with `unavailable` status + log warning

3. **Cert rotation** — `tools/mtls-ca/rotate.sh`:
   - Re-issue all leaf certs, deploy via rolling restart
   - Old certs remain valid for 24h overlap (grace period)

## Acceptance criteria

- [ ] `provision.sh` generates valid CA + leaf certs
- [ ] gRPC interceptor rejects connections without valid client cert
- [ ] `rotate.sh` re-issues leaf certs within 5 min
- [ ] 24h overlap grace period works
