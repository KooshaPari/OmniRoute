## Canonical release.yml pattern (cargo-dist style)

Every workflow (with the per-repo deviations noted above) follows this shape:

```yaml
name: Release
on:
  push:
    tags: ['v*.*.*']            # catches v1.2.3, v0.1.0-rc.1, etc.
  workflow_dispatch: …           # manual trigger with version input

permissions:
  contents: write                # needed to create GitHub Release + upload assets

concurrency:
  group: release-${{ github.ref }}
  cancel-in-progress: false      # never kill an in-flight release

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - checkout@v4 (fetch-depth: 0 for full git history)
      - rust-toolchain@stable (or setup-python@v5 for pheno-mcp-router)
      - Verify tag matches Cargo.toml version (fail-fast if drift)
      - cargo test --release --locked --all-features
      - cargo publish --dry-run --locked --all-features
      - cargo-cyclonedx --format json (SBOM)
      - cargo publish --locked --all-features (with CARGO_REGISTRY_TOKEN secret)
      - softprops/action-gh-release@v2 + attach SBOM files
```

All five files stay within the **80-line cap** (range 65–77).

## Workspace-member loop (for future cargo-dist multi-crate workspaces)

None of the 5 audited repos currently use a multi-crate workspace — each has `[workspace]` empty table or no `[workspace]` at all, declaring itself a standalone package. The pattern above does not need a per-crate loop. If a future repo (e.g., `pheno-runtime` if it grows into a workspace) needs one, the loop is:

```yaml
- name: Publish workspace members
  env:
    CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
  run: |
    for crate in $(cargo metadata --no-deps --format-version 1 | jq -r '.packages[].name'); do
      echo "::group::publish $crate"
      cargo publish -p "$crate" --locked || { echo "::error::publish failed for $crate"; exit 1; }
      echo "::endgroup::"
      sleep 30   # crates.io rate-limit courtesy
    done
```

This loop was **not** added to any of the 5 release.yml files because none currently publish multiple crates. Captured here for the future.

## SBOM policy (cross-reference)

The new `release.yml` files generate CycloneDX JSON via `cargo-cyclonedx` (Rust crates) or `cyclonedx-py` (Python). This is consistent with:
- ADR-040 (test coverage gates per tier) §3 — "every release carries an SBOM".
- `findings/2026-06-21-v15-L48-SBOM-diff.md` — CycloneDX chosen over SPDX for fleet consistency.
- `pheno-tracing/.github/workflows/sbom.yml` (pre-existing) — same tool, attached to every PR run; the new release.yml attaches the same artifact to the GitHub Release.
