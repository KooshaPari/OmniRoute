# Release Process (L26)

**Date:** 2026-06-22
**Status:** ACCEPTED
**Owner:** `orch-w1-a` (release coordination circle)
**Applies to:** All repos in the Phenotype monorepo and any standalone
crate that participates in coordinated releases.

This document is the canonical reference for cutting a Phenotype
release. It covers version semantics, the two supported release
flows, the tooling, the CHANGELOG contract, and recovery procedures.

## 1. Versioning

The monorepo uses **[SemVer 2.0.0](https://semver.org/)** with a
single shared version across all `pheno-*` / `phenotype-*` member
crates. The version is stored in **one place**:

- **Root `Cargo.toml` тЖТ `[workspace.package] version`** (`Cargo.toml:75-79`).
  Every member crate inherits this via `version.workspace = true`.

The top-level `VERSION` file at the repo root is **not authoritative**
and exists only for legacy tooling. The release script will warn if
`VERSION` disagrees with `[workspace.package] version`; do not update
`VERSION` by hand. The script does **not** touch `VERSION` тАФ only
`Cargo.toml` and `CHANGELOG.md`.

### SemVer rules for the monorepo

| Bump | When to use | Examples |
|------|-------------|----------|
| **MAJOR** | Public API of any `phenotype-*-sdk` crate has a breaking change. Per SDK Versioning Policy (`docs/sdk-versioning.md`). | `0.4.0 тЖТ 1.0.0` (first stable), `1.2.3 тЖТ 2.0.0` |
| **MINOR** | New backwards-compatible feature in any member crate; or any change in a `0.X.Y` cycle where `X > 0` is a breaking change. | `0.0.12 тЖТ 0.1.0`, `0.4.0 тЖТ 0.5.0` |
| **PATCH** | Backwards-compatible bug fix only. | `0.0.12 тЖТ 0.0.13` |

Pre-1.0 deprecation rules from `docs/sdk-versioning.md` apply: in
`0.X.Y` with `X > 0`, MINOR is breaking. In `0.0.Y`, anything can be
breaking.

### Tag format

Tags are **annotated** (`git tag -a`) and prefixed with `v`:

```
v0.0.13
v0.1.0
v1.0.0
v0.4.0-rc.1
```

The `release.yml` workflow refuses non-annotated tags.

## 2. Two release flows

There are two supported ways to cut a release. Both produce the
same end state: an annotated tag at origin, a `CHANGELOG.md`
section for the new version, and a GitHub Release with auto-generated
notes + a source tarball + an SPDX SBOM.

### Flow A тАФ Local (recommended for normal releases)

A human runs `scripts/release.sh` from a clean clone:

```bash
# Pre-flight
git checkout main && git pull --ff-only
# (working tree must be clean)

# Cut the release
./scripts/release.sh 0.0.13

# Optional: preview without mutating
./scripts/release.sh 0.0.13 --dry-run
```

The script performs steps (a)-(e) atomically:

| Step | Action | Mutates |
|------|--------|---------|
| (a) | Bumps `[workspace.package] version` in root `Cargo.toml` | `Cargo.toml` |
| (b) | Regenerates `CHANGELOG.md` from `git log` since the last tag, grouped by conventional-commit type via `git-cliff` (config: `cliff.toml`) | `CHANGELOG.md` |
| тАФ    | Commits the version bump + CHANGELOG as `chore(release): <version>` | one commit |
| (c) | Creates an annotated tag `v<version>` | one tag |
| (d) | Pushes the branch + the tag to `origin` | remote refs |
| (e) | Creates the GitHub Release via `gh release create` with auto-generated notes (falls back to the CHANGELOG section if `gh` notes are unavailable) | one release |

### Flow B тАФ CI-only (post-tag safety net)

If a tag is pushed without the local script (e.g. by a bot or
external automation), the `release.yml` workflow catches it:

1. Tag is pushed to `origin` matching `v[0-9]*.[0-9]*.[0-9]*`.
2. Workflow runs `scripts/release.sh <version> --ci`.
3. `--ci` skips steps (a)-(d) (already done) and only runs (e).
4. Workflow also attaches `phenotype-<ver>-source.tar.gz` and the
   SPDX SBOM to the release.

This is **idempotent**: re-running is safe; the only side effects
are (re-)uploading release assets and re-writing the step summary.

## 4. CHANGELOG contract

`CHANGELOG.md` is regenerated by `git-cliff` from conventional
commits since the previous tag. The grouping is defined in
`cliff.toml` at the repo root. Supported prefixes (anything else
falls into the "Other" group, or is skipped if marked `chore` /
`ci`):

- `feat:` тЖТ тЬи Features
- `fix:` тЖТ ЁЯРЫ Bug Fixes
- `doc:` / `docs:` тЖТ ЁЯУЪ Documentation
- `perf:` тЖТ тЪб Performance
- `refactor:` тЖТ тЩ╗я╕П Refactoring
- `test:` тЖТ ЁЯзк Testing

Commits with `BREAKING CHANGE:` in the footer or a `!` after the
type (`feat!:`) are flagged in the rendered output.

The template at `docs/CHANGELOG_TEMPLATE.md` is the human-readable
counterpart used for hand-written release notes; the script does
not consult it.

## 5. Pre-flight checklist

Before running `scripts/release.sh`:

- [ ] Working tree is clean (`git status` is empty).
- [ ] On `main` (or a `release/*` branch) and up-to-date with `origin`.
- [ ] All CI on `main` is green for the commit you intend to tag.
- [ ] `gh auth status` shows the correct account (must have
      `contents: write` and `packages: write` on the target repo).
- [ ] The target version is unused: `git tag --list 'v<version>'`
      returns empty.
- [ ] If a `VERSION` file exists, it's already in sync with
      `[workspace.package] version` in `Cargo.toml` (or you accept
      that the script will warn тАФ and you will fix the file in a
      follow-up commit).

## 6. Recovery procedures

### Wrong version pushed

```bash
# Delete the local + remote tag
git tag -d v0.0.13
git push origin :refs/tags/v0.0.13

# Delete the GitHub Release (the workflow does NOT auto-delete)
gh release delete v0.0.13 --yes

# Revert the version-bump commit
git revert <chore-release-commit-sha>
git push
```

### Workflow ran but release is broken

Re-run the workflow manually from the Actions UI; the script is
idempotent in `--ci` mode.

### CHANGELOG.md looks wrong

Edit `cliff.toml` to fix commit-parser rules, then re-run
`git-cliff --tag v<version> --output CHANGELOG.md` and amend the
`chore(release):` commit before pushing.

## 7. Permissions

`release.yml` requests the following GitHub token scopes (the
minimum needed today; `id-token` and `attestations` are reserved
for future SLSA provenance work):

| Scope | Reason |
|-------|--------|
| `contents: write` | Create the GitHub Release, upload source tarball + SBOM. |
| `id-token: write` | OIDC token for SLSA provenance (future). |
| `attestations: write` | Publish release attestations (future). |
| `packages: write` | Publish container artifacts (future). |

`cancel-in-progress` is **false** on the `release` concurrency
group: a release must run exactly once per tag. Re-running is safe
but we want a single canonical CI run per tag in history.
